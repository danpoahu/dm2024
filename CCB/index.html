Lead by PushPay API Research

Discover More ↔ CCB Integration Strategy • March 2026

Overview

Lead by PushPay (formerly Church Community Builder / CCB) has two separate APIs. The "Lead" mobile app does not have its own API — all CRM operations go through the legacy CCB API.

API Purpose Format Auth Need It?
CCB (ChMS) API People, notes, processes, custom fields XML REST HTTP Basic Auth ✓ YES
Pushpay API Payments & donations only JSON REST OAuth 2.0 ✗ NO

The CCB API base URL follows this pattern:
https://[yourchurch].ccbchurch.com/api.php?srv=[service_name]

API Capabilities

✓ Search for People

Service: individual_search — search by first name, last name, email, phone, address.

Also: individual_profile_from_id to look up a specific person by their CCB ID.

✓ Create New People

Service: create_individual — accepts first/last name, email, phone, address, campus, family, and more.

✓ Update People Records & Custom Fields

Service: update_individual — update any profile field including custom text, date, and pulldown fields.

Service: custom_field_labels — read custom field label definitions.

✓ Read Notes on a Person

Service: individual_notes — retrieve notes about an individual, with date filtering and private note options.

Note: Creating standalone notes is poorly documented, but notes can be added when placing someone in a process queue.

✓ Track Progress Through Processes (W1 / W2 / W3)

Services: process_list, queue_list, add_individual_to_queue, individual_queues, queue_individuals

Set up a "Discover More" process with queues for each week. Move people between queues via API as they complete steps. The DM app can read queue status back to show completion.

✗ Upload File Attachments

No API endpoint exists for uploading PDFs or documents to a person's profile. File management is limited to the CCB web interface.

✗ Webhooks for CRM Changes

No real-time notifications for people/process changes. You would need to poll the API. (Pushpay has webhooks but only for payment events.)

Integration Strategy: Custom Fields

Instead of uploading a PDF, we write the key results directly into CCB custom fields on the person's record. This makes the data instantly visible and searchable in Lead/CCB without staff needing to open a file.

Custom Field Mapping

CCB Field Label Example Value Source in DM
udf_text_1 DISC Score D/I discH + "/" + discL
udf_text_2 Top 3 Spiritual Gifts Leadership, Teaching, Mercy Top 3 from scored gifts
udf_date_1 DM Survey Completed 2026-03-09 Timestamp when surveys are submitted

Sync Flow: DM App → CCB

1
User completes surveys in Discover More.
DISC score (e.g. D/I) and top 3 spiritual gifts are calculated and saved to Firebase.
2
Search CCB for the person.
API call: individual_search with email or name. Get their individual_id.
3
Write custom fields to their CCB profile.
API call: update_individual with individual_id, udf_text_1 (DISC), udf_text_2 (gifts), udf_date_1 (date).
4
Add to process queue (optional W1/W2/W3 tracking).
API call: add_individual_to_queue to place them in the appropriate week's queue with a note.

W1 / W2 / W3 Progress Tracking

CCB's Process Queue system is ideal for this. Here's the setup:

CCB Setup (one-time, in the CCB admin)

  • Create a Process called "Discover More"
  • Add Queues: "Week 1", "Week 2", "Week 3", "Completed"
  • Assign queue managers (staff who oversee each step)

Two-Way Sync

  • DM → CCB: When a user completes W1 in the app, call add_individual_to_queue to move them into the W1 queue
  • CCB → DM: Call individual_queues to check which queues a person is in, then reflect that status in the DM app (show checkmarks for completed weeks)

Authentication & Setup

Getting API Credentials

  1. Log into your CCB admin panel
  2. Go to Settings → API
  3. Create a new API User
  4. Grant it access to the services you need: individual_search, update_individual, add_individual_to_queue, individual_queues, etc.
  5. API calls use HTTP Basic Auth with that username/password

Exploring Available Endpoints

Add &describe_api=1 to any API call to discover available parameters and response formats on your specific instance. This is the best way to find undocumented features.
